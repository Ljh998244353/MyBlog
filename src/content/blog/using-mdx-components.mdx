MDX 最有价值的地方，不是“语法更复杂”，而是它允许内容和少量结构化组件共存。

<Callout title="适用场景" tone="success">
  当一篇文章需要提示框、实验结论、对比表或者交互式示例时，MDX 比纯 Markdown 更合适。
</Callout>

## 什么内容值得做成组件

不是所有内容都应该组件化。一个简单判断是：它是否会在多篇文章里重复出现，而且每次都长得差不多。

- 提示框
- 风险说明
- 结论摘要
- 演示容器

## 保持文章可读性

使用 MDX 时要克制。组件只是表达手段，正文依然应该维持顺畅的阅读节奏。过多的卡片、按钮和装饰会削弱文章本身。

<Callout title="经验" tone="warn">
  如果一篇技术文章需要大量交互才能讲清楚，通常意味着结构还没整理好。
</Callout>
